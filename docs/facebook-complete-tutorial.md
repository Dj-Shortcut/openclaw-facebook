# Complete Facebook Messenger Tutorial

This guide explains the full Facebook/Meta side of running the OpenClaw
`facebook` channel: the Meta app, the Facebook Page identity, permissions,
Messenger rules, testing, public launch, and operational checks.

Use this guide when you need the whole picture. For the short install path, see
[setup.md](setup.md).

## 1. What This Integration Is

This plugin connects OpenClaw to **Facebook Page Messenger direct messages**.

The chain is:

```text
Person on Facebook/Messenger
-> Facebook Page inbox
-> Meta app webhook
-> OpenClaw facebook channel
-> Meta Send API
-> Messenger reply from the Page
```

The Facebook Page is the public identity people message. The Meta app owns the
webhook, permissions, app secret, and access review. OpenClaw receives webhooks
and sends replies through the Page access token.

This plugin does not currently implement:

- Instagram DMs
- Page comments
- comment-to-DM/private replies
- attachments or media downloads
- quick replies/templates
- automatic Page subscription setup
- general Meta automation outside Page Messenger DMs

## 2. Accounts And Identity

### Your personal Facebook account

You need a real Facebook account that can administer the Facebook Page and the
Meta app. Do not treat a personal profile as the public bot identity. Messenger
Platform messaging for a business flow requires a Page.

### Facebook Page

The Page is what users see in Messenger. Choose the Page name, avatar, category,
description, and public details carefully because they establish trust before
the user sends a message.

Recommended Page checklist:

- Page name matches the assistant or business users expect.
- Profile photo and cover image are not placeholders.
- About/contact info is filled in.
- Privacy policy URL is available for public or reviewed apps.
- Page admins are current and trusted.
- The Page is the exact Page whose ID is configured in OpenClaw.

### Meta developer account

Use Meta for Developers to create and manage the app:

```text
https://developers.facebook.com/apps/
```

Keep track of:

- app ID
- app secret
- app mode: Development or Live
- app roles: admin, developer, tester
- connected Business portfolio, if any
- permissions and access level

### Page-scoped sender IDs

When a person messages a Facebook Page, Meta gives your app a Page-scoped sender
ID for that person. The same person can have a different ID on another Page.

Use Page-scoped IDs only as channel identifiers. For logs, ledgers, and billing,
prefer a hashed/pseudonymous user key in the host runtime.

## 3. Meta App Setup

1. Open Meta for Developers.
2. Create or open the app for this Page.
3. Add the Messenger product if it is not already present.
4. Copy the app secret from **Settings > Basic**.
5. Connect the target Facebook Page in the Messenger settings.
6. Generate or copy a Page access token for that Page.
7. Configure the webhook callback URL.
8. Subscribe the Page to the webhook fields.

The app secret and Page token must belong to the same app/Page setup. A callback
can verify successfully but still reject real messages if the app secret used
for signature verification is from a different app.

## 4. Values OpenClaw Needs

| OpenClaw field | Where it comes from | Purpose |
| --- | --- | --- |
| `pageId` | Facebook Page details / Messenger settings | Identifies the Page receiving DMs |
| `pageAccessToken` | Messenger settings / Page token flow | Sends replies as the Page |
| `appSecret` | Meta app Settings > Basic | Verifies webhook signatures |
| `verifyToken` | You choose it | Verifies Meta's initial webhook challenge |
| `webhookPath` | Optional OpenClaw config | Lets multiple Pages use distinct routes |

Use a long random verify token. It is not generated by Meta.

```text
FACEBOOK_VERIFY_TOKEN=replace-with-a-long-random-string
```

## 5. OpenClaw Configuration

Minimal config:

```json5
{
  channels: {
    facebook: {
      enabled: true,
      name: "Leaderbot",
      pageId: "<FACEBOOK_PAGE_ID>",
      pageAccessToken: "<FACEBOOK_PAGE_ACCESS_TOKEN>",
      appSecret: "<FACEBOOK_APP_SECRET>",
      verifyToken: "<FACEBOOK_VERIFY_TOKEN>",
      dmPolicy: "pairing"
    }
  }
}
```

Environment variable alternative for the default account:

```text
FACEBOOK_PAGE_ID=
FACEBOOK_PAGE_ACCESS_TOKEN=
FACEBOOK_APP_SECRET=
FACEBOOK_VERIFY_TOKEN=
```

New installs should use `facebook` everywhere:

- plugin/channel id: `facebook`
- config key: `channels.facebook`
- default webhook path: `/facebook/webhook`
- env vars: `FACEBOOK_*`

Legacy `messenger`, `fb`, and `fbm` aliases exist only for compatibility.

## 6. Webhook Setup

Default callback URL:

```text
https://<gateway-host>/facebook/webhook
```

Meta verifies the callback with a GET request. OpenClaw responds only when:

- the path reaches the OpenClaw gateway
- `hub.mode=subscribe`
- `hub.verify_token` matches your configured verify token

For real POST events, Meta signs the request with `X-Hub-Signature-256`.
OpenClaw validates the raw body with `appSecret`.

Subscribe the Page to the webhook fields this plugin can safely accept:

- `messages`
- `messaging_postbacks`
- `message_reads`

The current plugin processes inbound text messages. Unsupported events should be
accepted by the webhook but ignored by the plugin.

## 7. Development, Testing, And Public Launch

### Development mode

In Development/Standard Access, testing is limited to users who have an allowed
role on the app, Page, or connected Business assets. This is normal. If an
ordinary public user cannot reach your bot in development mode, check roles and
access level before debugging OpenClaw.

Recommended development test accounts:

- app admin
- app developer
- app tester
- Page admin or Page tester where applicable

### Live mode and public users

For public usage, plan for:

- app Live mode
- Advanced Access where required
- App Review if users without roles will use the app
- Business Verification if Meta requires it for your access pattern
- privacy policy and data handling answers
- a screencast that proves the exact permission flow

If the app only sends and receives messages for your own Page, Meta's
requirements can be lighter than a multi-customer SaaS integration. If you want
other businesses or Pages to connect their assets, expect a stricter review path.

## 8. Permissions And Page Tasks

Messenger integrations commonly involve these permissions:

- `pages_show_list`
- `pages_manage_metadata`
- `pages_messaging`
- `pages_read_engagement`
- `business_management`

Depending on the app type and exact use case, Meta may require some permissions
only for advanced/public access or review. Do not request broad permissions
unless the product actually uses them.

The person or system user generating the Page token must have the Page tasks
needed to message and manage the Page integration. At minimum, check that the
user still has access to the Page and that the token has not expired.

## 9. Messenger Rules To Respect

### User-initiated conversations

Messenger Platform conversations must be initiated by the person. Do not use
this plugin as a cold outbound channel.

### Standard response window

When a person messages your Page, you can reply through the platform in the
allowed response window. For normal automated replies, design as if free-form
responses must happen within 24 hours of the user's last message.

If a reply fails outside the allowed window, this plugin reports the common
Graph API error as a Messenger response-window problem.

### Human agent exception

Meta documents a Human Agent feature for human support, using a `human_agent`
tag, in cases where a person needs help that cannot be resolved in the standard
messaging window. This plugin does not implement Human Agent handover/tagging.

### Automation and AI disclosure

Automated or AI-assisted Page chats may need to disclose that the user is
interacting with automation. For a public assistant, add clear wording at the
start of the conversation and after long gaps if required by local law or Meta
surface behavior.

### No spam or deceptive behavior

Do not send unsolicited promotions, misleading messages, hidden ads, or messages
that try to bypass Meta rules. Keep the assistant's identity and purpose clear.

### High volume and rate limits

Meta rate limits Messenger API usage. Meta's overview currently describes the
Messenger API app limit as:

```text
calls within 24 hours = 200 * number of engaged users
```

Meta also documents high-volume behavior where Page Inbox and sending can be
limited if a Page sends more than 40 messages per second or handles a very large
number of simultaneous conversations.

OpenClaw should still implement its own dedupe, rate limits, budget gates, and
tool policy. Meta limits protect Meta; they are not your cost-control layer.

## 10. Access Modes In OpenClaw

### Pairing mode

Recommended for private or early testing:

```json5
dmPolicy: "pairing"
```

Unknown senders get a short pairing code and their message is not processed by
the assistant until approved:

```bash
openclaw pairing list facebook
openclaw pairing approve facebook <CODE>
```

OpenClaw's shared pairing flow keeps pending codes short-lived and avoids
resending a new code for every spam message.

### Allowlist mode

Use this if you already know the Page-scoped sender IDs:

```json5
dmPolicy: "allowlist",
allowFrom: ["<PAGE_SCOPED_SENDER_ID>"]
```

### Open mode

Use only when anyone who messages the Page may enter the basic assistant flow:

```json5
dmPolicy: "open",
allowFrom: ["*"]
```

Open mode is only the channel entry point. It must not imply full Claw access.
Keep private files, memory, shell, deploy, GitHub mutations, machine starts,
browser automation, image generation, and high-cost tools behind host-level
trust and budget policy.

Public users may send personal, financial, authentication, or business-sensitive
information into Messenger. In open mode, those messages can be forwarded into
your OpenClaw host, model provider, logs, memory, and any enabled tools according
to your runtime configuration. Before enabling this for a public Page, publish a
privacy policy, disclose automated/AI handling where required, and decide what
data is retained, deleted, or shared with third-party providers.

For public or paid bots, read [cost-control-plan.md](cost-control-plan.md).

## 11. Cost And Tool Safety

Public Facebook traffic can become expensive when routed into a full assistant.
The transport plugin should not own billing, but the host runtime must enforce:

- unknown users do not reach model/tool turns
- approved users get a restricted default tool profile
- model/provider calls are logged per pseudonymous user key
- duplicate Meta deliveries do not create duplicate turns
- global Facebook daily budget cap
- user daily/session budget caps
- prepaid credits before high-cost features

Do not expose shell, deploy, cloud machine creation, private files, or GitHub
mutation tools to public Facebook users by default.

## 12. Multi-Page Setup

One plugin install can serve multiple Pages:

```json5
{
  channels: {
    facebook: {
      defaultAccount: "leaderbot",
      accounts: {
        leaderbot: {
          name: "Leaderbot",
          pageId: "<LEADERBOT_PAGE_ID>",
          pageAccessToken: "<LEADERBOT_PAGE_ACCESS_TOKEN>",
          appSecret: "<META_APP_SECRET>",
          verifyToken: "<LEADERBOT_VERIFY_TOKEN>",
          webhookPath: "/facebook/leaderbot",
          dmPolicy: "pairing"
        },
        support: {
          name: "Support",
          pageId: "<SUPPORT_PAGE_ID>",
          pageAccessToken: "<SUPPORT_PAGE_ACCESS_TOKEN>",
          appSecret: "<META_APP_SECRET>",
          verifyToken: "<SUPPORT_VERIFY_TOKEN>",
          webhookPath: "/facebook/support",
          dmPolicy: "pairing"
        }
      }
    }
  }
}
```

Each account should have:

- its own Page ID
- its own Page token
- a matching verify token
- a distinct webhook path when sharing one gateway
- its own budget/tool policy in the host runtime if public

## 13. Production Checklist

Before a real public launch:

- Meta app is in the intended mode.
- Page is connected and has correct public identity.
- Page token is long-lived or managed through a reliable system-user flow.
- Webhook GET verification works.
- Webhook POST signature validation works.
- Page is subscribed to the required webhook fields.
- `dmPolicy` is deliberately chosen.
- Public mode has host-level tool restrictions.
- Cost ledger and budget gate exist before high-cost model/tool access.
- Privacy policy and data deletion/contact flow are available.
- Automated/AI disclosure is clear where required.
- Owner knows how to inspect logs and pairing requests.
- Owner knows how to disable the channel quickly.

## 14. Smoke Test

1. Start or deploy the OpenClaw gateway.
2. Verify the Meta callback URL in the app dashboard.
3. Send a Messenger DM to the Page from an allowed test account.
4. If using pairing mode, approve the code.
5. Send another text message.
6. Confirm OpenClaw replies.
7. Stop the gateway and confirm Meta retries do not cause duplicate assistant
   turns after restart/recovery.

Expected:

- webhook verification succeeds
- inbound text messages reach OpenClaw
- unsupported events do not crash the plugin
- duplicate message IDs are skipped
- replies use the correct Page access token
- no separate active `messenger` channel appears for new installs

## 15. Troubleshooting

### Callback URL cannot be verified

Check:

- URL is public HTTPS
- path is `/facebook/webhook` or your configured `webhookPath`
- gateway is running
- verify token matches exactly
- proxy keeps query parameters intact

### Webhook verifies but messages do not arrive

Check:

- Page is subscribed, not only the app object
- sender has access under app mode/access level
- message is sent to the connected Page
- webhook fields include `messages`
- Meta app/Page roles allow the tester
- OpenClaw logs show the incoming request

### Messages arrive but OpenClaw does not answer

Check:

- `dmPolicy` is not blocking the sender
- pairing code has been approved if in pairing mode
- host runtime did not block the message by budget/tool policy
- the event is a text message supported by this plugin

### Replies fail

Check:

- Page token belongs to the same Page
- token is valid and has the needed access
- app has Messenger permissions required for the use case
- conversation is inside the allowed response window
- Page/user is not unavailable or blocked
- app secret matches the app that owns the webhook

### European Messenger warning

Some users in Europe may see a Messenger UI warning about new messaging rules.
If no webhook reaches OpenClaw, the user did not enter the Page conversation.
If the webhook reaches OpenClaw but no reply is sent, continue with token,
policy, and response-window debugging.
